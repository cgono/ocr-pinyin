## What Went Well

- **Adapter/protocol pattern paid off immediately.** The `OcrProvider` protocol established in Story 1.4 enabled a complete provider swap in Story 1.7 (Textract → Google Cloud Vision) with zero changes to the service layer, route handlers, or schemas. One new adapter file and one factory update.
- **CI quality gates established before feature expansion.** Contract tests in Story 1.2 caught response envelope drift during Stories 1.4 and 1.5 when the schema was extended.
- **Clean architecture layering maintained throughout.** The api / schemas / services / adapters / core / middleware split was consistently respected across all 7 stories.
- **Sprint change proposal process worked.** Story 1.7 was introduced via an approved sprint change proposal (`sprint-change-proposal-2026-03-22.md`), giving it clear scope, rationale, and acceptance criteria.
- **Process was predictable.** Overall team cadence and story execution were smooth; the foundation epic delivered stable contracts for Epic 2 to build on.

## Challenges

### 1. OCR Provider Not Validated Against Target Content

**What happened:** AWS Textract was named in the architecture and implemented in Story 1.4. During integration testing, it was discovered that Textract does not support Chinese script — the core content type of the product. This required a course correction (Story 1.7) that was unplanned.

**Root cause:** No provider validation was done against real Chinese content before the architecture committed to Textract. Familiarity gap — it's a well-known AWS service and Chinese support was assumed, not verified.

**Impact:** One unplanned story added mid-epic; Story 1.4's OCR adapter had to be largely replaced.

### 2. Wrong-Scope Story 1.6 Implementation

**What happened:** A previous implementation attempt for Story 1.6 implemented a LangGraph orchestration migration instead of the specified OpenAPI spec exposure + Bruno developer collection. The commit was reverted and the correct implementation delivered separately.

**Root cause:** A personal learning goal (exploring LangGraph) was not communicated through the sprint change process. It was pursued informally and surfaced in the implementation rather than being planned, scoped, and either approved or deferred.

**Impact:** One reverted commit; context switch and redo for Story 1.6.

---

## Lessons Learned

1. **Validate external providers against real target content before architecture commits.** For any provider central to core functionality, a quick spike — even manual testing — is required before it's named in architecture or story specs.

2. **Personal learning goals and technology substitutions must go through the sprint change proposal process.** If a technology or approach changes, it needs a formal proposal with scope and rationale — not an informal implementation. The sprint change process exists and worked well for Story 1.7; use it consistently.
